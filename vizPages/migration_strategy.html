Migration to ArcGIS Enterprise  •  Proposal, Diagrams, and Timeline (2019)

ArcGIS Enterprise combines the four software components to develop a cohesive system, including ArcGIS Web Adaptor, Portal for ArcGIS, ArcGIS Server and the ArcGIS Data Store.

Note, this is a mock proposal developed in 2019. ArcGIS Enterprise may have deployed new versions/features following this proposal.

Portal for ArcGIS (Web Tier)

The Portal component of ArcGIS Enterprise allows you to share and secure your organization’s geospatial content and is known as the web application component of Enterprise. It allows you to make and share maps, as well as analyze data to find trends. Portal contains two main features: A web front end and an infrastructure back end. This design is meant to support users’ interaction and experience in Enterprise portal. Portal connects to your ArcGIS Server, which also connects to your administrator in Portal. Portal has its own database where you can hold non-spatial (or spatial) data – and is ultimately configured in coding languages like Python. Note that for Portal to work, Web Adaptor must also be connected.

ArcGIS Server (App Tier - Hosting Server)

The ArcGIS Server component allows you to publish services and share maps from your organization. Server hosts layers, maps and services that your organization uses to power their GIS applications. Server is mainly used for visualization and analysis and serves as a conduit between database and web interfaces – mainly creating REST interfaces. As a side note, Server was developed during the ArcGIS Desktop era and therefore is compatible with Desktop. ArcGIS Server has five main capabilities: GIS Server, Image Server, GeoEvent Server, GeoAnalytics Server and Business Analyst Server.

ArcGIS Data Store (Data Tier – Relational & Tile Cache)

The Data Store component holds your organization’s data – specifically the spatial data that is shared in ArcGIS Enterprise Portal. You can have many GIS data servers connected to one Enterprise. The Data Store consists of relational (for storing hosted data and outputs from analysis tools), tile cache (for storing 3D tile caches for 3D visualization), and spatiotemporal databases (for storing large volumes of events). Some of the main functions of the Data Store are publishing a high volume of hosted feature layers, publishing layers to portal, archiving real time observation data, developing automated backup data, performing analysis and adding data to map applications.

ArcGIS Web Adaptor

The Web Adaptor component is the application that runs in your current website and sends the information to the ArcGIS Server, and therefore allows ArcGIS Server to work with 3rd party web servers. Additionally, it leverages your web server and provides more control to your site. It is also a reverse proxy for Web GIS access that enables web-tier authentication. This component is required for ArcGIS Portal and is optional for ArcGIS Server.

Modernization and Configuration

In order to modernize the current GIS data and implementation structure for the City of Washington, it would be best to evolve to a distributed web GIS system that is “configured first” with a consistent architecture system. The first plan of action will be to integrate all systems to ArcGIS, so they are all consistently served through the ArcGIS platform, and ultimately making the implementation and configuration process more fluid. Additionally, since the City of Washington’s users consist of GIS staff editors, GIS staff viewers, and public users, we will need to design a distributed web GIS design through ArcGIS Enterprise. A distributed web GIS design strategy will allow users within the organization to hold privileges and access to my databases, while allowing us to share our information securely to the public.

High priority upgrades to the current architecture

We will need to integrate into the base ArcGIS Enterprise system which consists of two ArcGIS Web Adaptors (one for ArcGIS Server to publish maps and reports, and one for Portal to display the web applications), an ArcGIS Data Store to host the city’s data repository, ArcGIS Portal to develop mapping applications and ArcGIS Server, which will need to be upgraded from the city’s current version as a hosting mechanism for the applications and data.

Medium priority modern add-ons to the ArcGIS Enterprise system

In order to address the needs of quick business analytics and mobile accessibility for our users I propose adding on the ArcGIS Workforce application and the GeoAnalytics Server.




Diagrams












System Requirements & Corresponding Solution

Create internal user/department groups where internal users can share content and maps between users and departments

Solution: ArcGIS Enterprise would work in this scenario as a cohesive system, as it provides content management capabilities and sharing mechanisms across organizations. Enterprise allows for more complex GIS data sharing with the Data Store and Portal (which define who can see what), and publishes them directly to those sources via ArcGIS Server.

Create different web viewers per department showing data pertaining to that department

Solution: ArcGIS Portal will supply the actual map viewing app, while the ArcGIS Data Store will provide the different data pertaining to each department.

Edit and publish data for both current and new GIS data layers

Solution: ArcGIS Server, which is built into ArcGIS Enterprise, would publish services, such as maps, feature data layers and geoprocessing to our users. Again, this is part of the ArcGIS Enterprise base deployment functionality, so it is within a streamlined system for deploying mapping technologies to our users. Since the current City of Washington structure already utilizes ArcGIS Server, we will just need to upgrade the ArcGIS Server used to an updated version which includes Web Adaptor, Portal, Server and Data Store.
Replace Adobe Flex MapViewer with a modern web GIS technology

Solution: ArcGIS Portal, which is built into ArcGIS Enterprise, would serve as the web GIS technology in this scenario, as it is a web application that we can pull from our new integrated architecture. It pulls the data from ArcGIS Data Store, serves it up on the ArcGIS Server and utilizes the Web Adaptor to display it to a third party, like our public users.

Collect mobile data from the field

Solution: ArcGIS Workforce. This platform links with ArcGIS Enterprise and allows you to capture field data in real time from a smart device. Additionally, since it links directly with ArcGIS Enterprise this loops into our cohesive system and allows for minimal error through migration.

Build information products to share with the public users

Solution: ArcGIS Server provides the analytics to be shared with public users via the GeoAnalytics Server (which is one of the five servers within ArcGIS Server). ArcGIS server will pull the data from ArcGIS Data Store in the Spatiotemporal Big Data Store which is from the GeoAnalytics Server.

Migration Strategy

The City of Washington’s migration process will transition the mixed GIS/web based application architecture into the consistent ArcGIS Enterprise environment to limit redundancy and promote data consistency and collaboration. The migration process will need to allow legacy GIS and web applications to be accessed throughout the infrastructure update. APIs can allow for direct connectivity and synchronization between the legacy applications and the updated ArcGIS Enterprise architecture throughout the migration. This will ensure that there is the necessary data replication in the new and old environments, and that data is available to the city when it is needed. The API that links the two systems will be disconnected once the legacy applications migrate to the updated ArcGIS Enterprise architecture system. The timeline shows that the timeline for implementation is estimated to take one year and eight months.
